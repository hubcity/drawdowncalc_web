## 2025-03-09 - [Prevent Hydration Mismatches When Sanitizing HTML in Next.js]
**Vulnerability:** Adding DOMPurify to React components directly can cause SSR Vulnerability Bypasses and Hydration mismatches.
**Learning:** Client-only sanitization logic (e.g., `typeof window !== 'undefined' ? DOMPurify.sanitize(...) : rawHtml`) completely defeats the purpose of the sanitizer in a Next.js application. XSS payloads in SSR frameworks are executed when the initial server HTML is parsed by the browser. By skipping sanitization on the server, the malicious payload would still be delivered and executed *before* client-side React hydration (and DOMPurify) ever runs. Additionally, `DOMPurify` is an HTML sanitizer, not a CSS sanitizer. It safely escapes characters like `<` and `>` into HTML entities (`&lt;`, `&gt;`). However, browsers **do not decode HTML entities inside `<style>` tags**. If the CSS ever utilizes child selectors (e.g., `>`) or other specific characters, `DOMPurify` will escape them, outputting invalid CSS and silently breaking the application's styling.
**Prevention:** Avoid adding DOMPurify to Shadcn UI `Chart` configurations or components where inputs are typically static or developer-controlled, as it is considered 'security theater' and unnecessary. Use security headers in Next.js config to mitigate general web vulnerabilities instead.
